1. Introduction

1.1 Background

SmartTrade Africa Ltd is a rapidly growing digital commerce company based in Tanzania that plans to launch a modern e-commerce platform allowing customers to buy products online via both mobile and web applications. The company has identified critical trust and security challenges including fake customer accounts, online payment fraud, unauthorized access attempts, identity theft, customer distrust toward online payments, and data privacy concerns.

In the Tanzanian and broader East African e-commerce landscape, trust remains one of the biggest barriers to adoption. Many potential customers are hesitant to provide personal information or make online payments due to fears of fraud and data misuse. SmartTrade Africa Ltd recognizes that addressing these concerns is not merely a technical requirement but a fundamental business necessity. Without customer trust, even the most feature-rich platform will fail to achieve meaningful adoption and transaction volume.

The platform is designed to serve both urban and peri-urban customers across Tanzania, where smartphone penetration is growing rapidly but digital literacy varies widely. This diversity requires the platform to be intuitive enough for first-time online shoppers while providing the advanced security features that experienced users demand. The solution must bridge this gap by making security transparent and trust visible at every interaction point.

1.2 Problem Statement

Traditional e-commerce platforms face significant challenges in building and maintaining customer trust. Security breaches, identity theft, and payment fraud have eroded consumer confidence in online transactions. In the East African context, additional challenges include limited familiarity with online payments, concerns about data privacy across borders, and the prevalence of mobile money fraud. SmartTrade Africa Ltd requires a comprehensive solution that addresses these challenges through the integration of trusted computing technologies, biometric authentication, secure payment processing, and trust management principles.

The core tension in e-commerce trust management is that users must provide sensitive information (payment details, personal data, delivery addresses) before they have established trust in the platform. This "trust paradox" requires the platform to demonstrate its trustworthiness proactively through visible security indicators, transparent policies, and verifiable security mechanisms. The platform addresses this by making trust visible through multiple layers: biometric authentication badges, secure payment gateway logos, and clearly communicated security policies that users can review before committing to a transaction.

1.3 Project Objectives

Each objective addresses a specific dimension of the trust and security challenge:

  • Develop a secure e-commerce web application with comprehensive features: Build a full-featured platform that meets the operational needs of both customers and administrators while embedding security at every layer of the architecture rather than treating it as an add-on.
  • Integrate Trusted Computing Platform (TCP) concepts for secure session and device management: Apply hardware-rooted trust concepts to the web application context, ensuring that sessions are securely managed and devices can be verified. This includes secure token handling, session integrity monitoring, and device fingerprinting.
  • Implement WebAuthn-based fingerprint biometric authentication: Replace or augment password-based authentication with phishing-resistant biometric authentication using the FIDO2/WebAuthn standard, which uses public-key cryptography to eliminate the risk of credential theft.
  • Integrate multiple simulated and real payment APIs (Stripe, PayPal, Mock): Provide payment flexibility while maintaining security through the Strategy Pattern, which allows the system to switch between payment gateways without changing core business logic. The Mock gateway enables testing without financial risk.
  • Apply trust management principles throughout the system: Embed trust-building mechanisms at every user touchpoint, from registration through checkout to post-purchase support, creating a consistent narrative of security and reliability.
  • Improve customer trust and user confidence through transparent security indicators: Display visible trust signals (security badges, authentication status, secure payment labels) throughout the user interface so that users can continuously verify the security of their session.
  • Apply TAM/UTAUT concepts to improve technology adoption and usability: Design the user experience based on established technology acceptance theories to ensure that security features do not come at the cost of usability, addressing both perceived usefulness and perceived ease of use.
  • Implement secure software development practices (CSRF, XSS, SQL injection prevention): Follow OWASP guidelines and Laravel security best practices to protect against the most common web application vulnerabilities, ensuring the platform is resilient against attacks.

1.4 Scope

The application covers user registration and authentication (including biometrics), product catalog management, shopping cart and wishlist functionality, secure checkout with multiple payment gateways, order management, customer reviews and ratings, admin dashboard with analytics, and comprehensive trust and security features. The scope deliberately excludes native mobile application development (which is recommended as a future enhancement) and focuses on delivering a responsive web application that works seamlessly across desktop and mobile browsers. The biometric authentication leverages browser-native WebAuthn APIs, which means it works on any device with a compatible browser and biometric sensor without requiring a separate mobile application.

2.2 System Architecture Diagram

The application follows Laravel's MVC (Model-View-Controller) architecture with a service layer for payment gateway abstraction. The MVC pattern was chosen because it provides a clean separation of concerns: Models handle data and business logic, Views handle presentation, and Controllers handle request flow. This separation is critical for security because it allows each layer to implement its own security measures independently. For example, Models use Eloquent ORM to prevent SQL injection, Views use Blade's automatic escaping to prevent XSS, and Controllers use middleware and validation to enforce access control.

The service layer pattern (specifically the Strategy Pattern for payment gateways) adds an additional level of abstraction that decouples the payment processing logic from the controllers. This means new payment gateways can be added by implementing a single interface (PaymentGatewayInterface) without modifying any existing code — a textbook example of the Open/Closed Principle from SOLID design.

The request flow through the architecture:

5. Payment API Integration

5.1 Multi-Gateway Architecture

The payment system uses the Strategy Pattern via PaymentService and PaymentGatewayInterface, allowing seamless switching between gateways. The Strategy Pattern is particularly well-suited for payment processing because it allows the system to define a family of algorithms (payment gateways), encapsulate each one, and make them interchangeable. The client code (PaymentController) interacts only with the interface, never with specific gateway implementations. This means that adding a new payment gateway requires only creating a new class that implements the interface — no changes to controllers, views, or business logic are needed.

The PaymentGatewayInterface defines three key methods that every gateway must implement:

  • processPayment(Order $order, User $user): PaymentResult — Initiates the payment transaction and returns a result object with status, transaction reference, and redirect URL if applicable.
  • verifyPayment(string $transactionRef): PaymentStatus — Verifies the status of a transaction using the gateway's lookup API to ensure the payment was completed successfully before confirming the order.
  • refundPayment(Payment $payment): PaymentResult — Processes a refund through the gateway's refund API, which is used by the admin panel for order disputes and cancellations.

The PaymentService acts as a factory and orchestrator. When a user selects a gateway during checkout, the service resolves the appropriate gateway implementation based on the selected key (e.g., 'stripe', 'paypal', 'mock'). This resolution can be configured dynamically through the database or configuration, allowing administrators to enable or disable specific gateways without code changes.

6. Fingerprint Biometric Authentication (WebAuthn)

6.1 Technology: WebAuthn / FIDO2

The application implements the W3C Web Authentication API (WebAuthn) standard using the lbuchs/webauthn PHP library. WebAuthn is a phishing-resistant authentication standard that allows users to authenticate using biometric sensors (fingerprint, face recognition), platform authenticators (Windows Hello, Apple Touch ID), or external security keys.

Unlike traditional password-based authentication, WebAuthn uses asymmetric (public-key) cryptography. When a user registers a biometric credential, their device generates a public-private key pair. The private key never leaves the device — it is stored in secure hardware (TPM on Windows, Secure Enclave on Apple devices, TrustZone on Android). The public key is sent to the server and stored in the database. During authentication, the server sends a challenge (random bytes), and the device signs this challenge with the private key. The server verifies the signature using the stored public key. This means that even if an attacker gains access to the server database, they cannot impersonate the user because the private key never leaves the user's device and cannot be extracted from the secure hardware.

WebAuthn is fundamentally phishing-resistant because the authentication response is scoped to the specific origin (domain). A WebAuthn credential registered for hesmb.com will not work on hesmb-login.com because the browser validates the origin as part of the authentication ceremony. This eliminates the entire class of phishing attacks that plague password-based authentication.

6.2 Key Features

  • Biometric Registration: Users enroll their fingerprint via browser's native credential management API. The browser handles the interaction with the platform's biometric hardware (Windows Hello, Touch ID, Android fingerprint), ensuring that the biometric data never leaves the device. What is transmitted to the server is solely the public key and attestation data.
  • Biometric Authentication: One-tap fingerprint login after initial password authentication. The user can authenticate with a single fingerprint scan, eliminating the need to remember or type passwords.
  • Credential Management: Users can view, add, and remove enrolled biometric credentials from their profile page. Each credential is identified by its creation date and a friendly name, allowing users to manage multiple devices.
  • Remote Setup: Token-based biometric enrollment links for device setup. The system can generate a unique, time-limited token that allows a user to register a biometric credential from a different device. This is useful when setting up a new phone or computer without requiring a password entry on that device.
  • Phone Verification: OTP-based phone verification before biometric enrollment. To ensure that biometric enrollment is protected against unauthorized access, users must verify their phone number via a one-time password (OTP) before they can register a fingerprint. This adds an additional layer of security to the enrollment process.
  • Device Support Check: Automatic detection of WebAuthn support on the user's device via JavaScript. If the user's browser or device does not support WebAuthn (e.g., older browsers), the biometric options are hidden and the user is informed that biometric authentication is not available on their current device.

6.3 Implementation Details

The biometric system works through a challenge-response protocol that involves both client-side JavaScript and server-side PHP code. The complete cryptographic flow:

=== Registration Flow === 1. Server generates a random 32-byte challenge and creates a WebAuthn credential creation options object 2. Browser's navigator.credentials.create() is called, which prompts the user for fingerprint/biometric verification 3. Device's secure hardware generates an asymmetric key pair (ECDSA P-256 or RSA) and returns the public key, credential ID, and attestation 4. Server receives the credential and verifies the attestation (checks that the authenticator is genuine) 5. Server stores the credential ID, public key, and counter value in the 'credentials' table, associated with the user === Authentication Flow === 1. Server generates a random 32-byte challenge and retrieves the user's stored credential IDs 2. Browser's navigator.credentials.get() is called with the challenge and credential IDs 3. Device finds the matching credential, prompts for fingerprint, and signs the challenge with the private key 4. Server verifies the signature using the stored public key, checks the counter value for cloned credentials, and authenticates the user

The PHP library lbuchs/webauthn handles the low-level cryptographic operations including: CBOR decoding of the attestation object, signature verification using the COSE key format, and counter-based clone detection. The counter is an anti-cloning mechanism: each time the authenticator is used, its internal counter increments. If the server receives a counter value lower than the stored value, it indicates that the credential may have been cloned and the authentication is rejected.

7.2 Session Security

The session driver is set to database for persistence and auditability. Unlike file-based sessions (Laravel's default), database sessions persist across server restarts and can be centrally managed in a load-balanced environment. The session lifetime is configured to 120 minutes via the SESSION_LIFETIME environment variable, after which the user is forced to re-authenticate.

Session fixation is prevented by regenerating the session ID after every successful login (Laravel's regenerate() method on the session). This ensures that any session ID set by an attacker before login is invalidated when the user authenticates.

Each session record in the sessions database table stores the following metadata for security auditing:

  • User ID: Links the session to the authenticated user for accountability
  • IP Address: Captured for geographic analysis and anomaly detection (e.g., a session that suddenly appears from a different country)
  • User Agent: Provides browser and operating system information for device identification
  • Last Activity: Timestamp of the most recent request, used to enforce idle timeout
  • Payload: Encrypted session data stored as a BLOB

The session security implementation also includes middleware that validates the session integrity on every request. If the user's IP address or user agent changes significantly between requests (indicating possible session hijacking), the session can be invalidated and the user logged out.

7.3 Token-Based Security

Token-based security provides proof of authorization for specific operations. Each token type has a distinct purpose and security profile:

  • CSRF Tokens: Every form includes a CSRF token validated by Laravel's VerifyCsrfToken middleware. The token is generated per-session and automatically included in every form via the @csrf Blade directive. For AJAX requests, the token is read from a <meta> tag and sent in the X-CSRF-TOKEN header. The token is tied to the user session and expires when the session expires.
  • Email Verification Tokens: Laravel's signed URLs provide tamper-proof verification links. The URL includes a signature (HMAC-SHA256) computed using the application key. Any modification to the URL invalidates the signature, causing the verification to fail. The URLs also include an expiration timestamp (default 60 minutes) to limit the window of opportunity.
  • Password Reset Tokens: Hashed tokens stored in the password_resets table. The token is generated using Str::random(60) and hashed with bcrypt before storage, ensuring that even if the database is compromised, tokens cannot be reused or generated. Tokens expire after 60 minutes.
  • Biometric Setup Tokens: 64-character random tokens (generated with bin2hex(random_bytes(32))) for remote WebAuthn enrollment. These tokens allow a user to register a biometric credential from a different device by entering the token. The token is one-time-use and expires after a configurable period (default 30 minutes).
  • API Tokens: Session-based authentication for AJAX requests. The session cookie is automatically included in same-origin AJAX requests, and the CSRF token is sent in the request header. No separate API token mechanism is needed because the session provides authentication and the CSRF token provides authorization.

9.1 Data Security

Data security protects the confidentiality, integrity, and availability of user data. The platform implements multiple layers of data protection, each addressing a specific threat vector:

Even if the password database is compromised, attackers cannot reverse the hashes to recover plaintext passwords. Each hash also includes a unique salt, preventing rainbow table attacks.

9.2 Authentication Security

Authentication is the first line of defense against unauthorized access. The platform implements defense-in-depth for authentication, meaning that multiple independent security controls must be bypassed for an attacker to gain access:

  • Rate Limiting: 5 login attempts per minute using Laravel's RateLimiter. This prevents brute-force password guessing attacks by temporarily locking the account after too many failed attempts. The rate limit applies per email address and per IP address, preventing both targeted attacks on specific accounts and distributed attacks across multiple accounts.
  • Session Regeneration: Session ID regenerated after successful login. This prevents session fixation attacks, where an attacker tricks a user into using a session ID chosen by the attacker. By generating a new session ID after authentication, any pre-existing session fixation is invalidated.
  • Session Timeout: 120-minute session lifetime with forced re-authentication. This limits the window of opportunity for an attacker who gains access to an unattended session. The timeout applies to both idle sessions (no activity for 120 minutes) and absolute sessions (total session lifetime).
  • Email Verification: Required before accessing protected features. This ensures that the email address provided during registration actually belongs to the user, preventing registration with fake or mistyped email addresses. The verification link is a signed URL that expires after a configurable period.
  • Biometric Authentication: Phishing-resistant WebAuthn/FIDO2 protocol. Unlike passwords, biometric credentials cannot be stolen through phishing because WebAuthn responses are scoped to the origin domain. Even if a user is tricked into visiting a fake login page, their biometric credential will not authenticate because the browser detects the domain mismatch.
  • Password Confirmation: Re-authentication required before sensitive actions. High-risk actions (such as changing email, password, or enrolling biometric credentials) require the user to re-enter their password, even if they are already logged in. This prevents an attacker who has gained temporary access to an unattended session from making permanent changes to the account.

9.3 Application Security

Application security protects against vulnerabilities in the code itself. The OWASP Top 10 provides a framework for the most critical web application security risks, and the platform addresses each relevant risk:

VulnerabilityPrevention MethodTechnical Explanation
SQL InjectionEloquent ORM with parameterized queries (prepared statements); no raw SQL queriesSQL injection occurs when user input is directly concatenated into SQL queries, allowing attackers to execute arbitrary database commands. Eloquent ORM uses PDO prepared statements which separate SQL structure from data — user input is always treated as data, never as executable SQL. The application strictly avoids raw SQL queries (DB::raw(), whereRaw()) in all production code paths.
Cross-Site Scripting (XSS)Blade's automatic {{ }} escaping; Content-Security-Policy headersXSS allows attackers to inject malicious JavaScript into web pages viewed by other users. Laravel's Blade template engine automatically escapes all output using {{ }} which converts HTML special characters to their entity equivalents (e.g., <script> becomes &lt;script&gt;). When unescaped output is needed ({!! !!}), it is used only with trusted, server-generated content.
CSRFCSRF token on every form via @csrf directive; token validation middlewareCross-Site Request Forgery tricks authenticated users into submitting unwanted actions. Every form and state-changing AJAX request includes a unique CSRF token that is validated by Laravel's VerifyCsrfToken middleware. The token is tied to the user's session and expires after a configurable timeout, preventing attackers from forging requests even if they know the token format.
Input ValidationForm Request validation rules; server-side validation for all inputsAll user input is validated server-side before processing, using Laravel's validation rules. This prevents malformed data from reaching the database or business logic. Validation includes type checking, length limits, format validation (email, URL, phone), and business rule validation (stock availability, price consistency). Client-side validation is used only for user experience — server-side validation is the authoritative check.
Mass AssignmentProtected $fillable/$guarded properties on all modelsMass assignment vulnerabilities allow attackers to set arbitrary model attributes by passing unexpected field names in requests. Each Eloquent model explicitly defines which attributes can be mass-assigned via the $fillable property. This ensures that even if an attacker adds extra fields to a request (e.g., is_admin=1), those fields are ignored by the model's create() and update() methods.
Path TraversalStorage facade with proper disk configuration; no direct filesystem accessPath traversal attacks attempt to read or write files outside the intended directory by including ../ sequences in file paths. The application uses Laravel's Storage facade which restricts file operations to configured disks. File uploads are stored with hashed filenames (generated by $file->store()), preventing attackers from guessing or traversing to other files.

9.4 Payment Security

Payment security is critical because it involves both financial assets and sensitive personal data. The platform follows the principle of data minimization — never collect or store data that is not absolutely necessary for the transaction:

  • No credit card numbers stored in the database or logs: Credit card numbers (Primary Account Numbers, PANs) are never transmitted to or stored on the platform's servers. All card data entry happens on the payment gateway's PCI-compliant pages, which use iframes or redirects to ensure that the merchant server never sees the full PAN. This eliminates the risk of a database breach exposing financial data.
  • All payment processing via gateway-hosted pages (Stripe Checkout, PayPal): The platform uses Stripe Checkout and PayPal Standard, which are hosted payment pages that handle the entire payment flow. The platform never directly handles card data, avoiding PCI DSS SAQ scope for the most stringent requirements. The platform only receives a payment token or transaction ID after processing is complete.
  • Transaction status verified end-to-end before order confirmation: After a user completes payment on the gateway's page, the platform verifies the transaction status by calling the gateway's verification API (e.g., Stripe's checkout.session.retrieve()). This prevents users from simulating a successful payment without actually completing the transaction.
  • Database transactions ensure atomicity (all-or-nothing order creation): Order creation involves multiple database operations: creating the order record, creating order items, updating product stock, clearing the cart, and creating the payment record. These operations are wrapped in a database transaction (DB::transaction()) so that if any operation fails, all changes are rolled back. This prevents partial orders where stock is decremented but no order is created, or where a payment is recorded but no order exists.
  • Payment records track complete audit trail with gateway references: Each payment record stores the gateway transaction ID, the amount processed, the gateway used, timestamps for each status change, and the user who performed the action. This provides a complete audit trail for financial reconciliation, dispute resolution, and fraud investigation.

10. TAM / UTAUT Integration

Technology acceptance theories explain why users adopt or reject new technologies. For an e-commerce platform, understanding these factors is critical because even the most secure system will fail if users find it difficult to use or do not see its value. This section explains how the platform implements the key constructs of the Technology Acceptance Model (TAM) and the Unified Theory of Acceptance and Use of Technology (UTAUT).

10.1 Technology Acceptance Model (TAM)

Developed by Fred Davis in 1989, TAM is one of the most widely used models in information systems research. It proposes that two primary factors determine technology acceptance: perceived usefulness (the degree to which a person believes that using the system will enhance their performance) and perceived ease of use (the degree to which a person believes that using the system will be free of effort). The application is designed following the Technology Acceptance Model (TAM), focusing on these two key constructs:

a) Perceived Usefulness (PU)

The application helps customers complete shopping activities efficiently:

  • Fast Checkout: Streamlined multi-gateway checkout process with minimal steps
  • Quick Payment Confirmation: Real-time payment status updates and instant order confirmation notifications
  • Efficient Product Search: Category filtering and keyword search with paginated results
  • Helpful Notifications: Real-time database notifications for order confirmations and updates
  • Product Discovery: Category browsing, search, and wishlist management

b) Perceived Ease of Use (PEOU)

The system is simple and easy to use:

  • Clear Navigation: Bootstrap 5 navbar with desktop/mobile responsive layouts, breadcrumb-style back button, and offcanvas mobile menu
  • Simple Registration Process: Minimal fields (name, email, password, phone) with clear validation messages
  • Easy Fingerprint Login: One-tap biometric authentication after initial setup
  • User-Friendly Interface: Tailwind CSS with consistent design, hover effects, loading animations, and toast notifications
  • Responsive Design: Mobile-first approach with offcanvas navigation and adaptive grid layouts

10.2 Unified Theory of Acceptance and Use of Technology (UTAUT)

Proposed by Venkatesh et al. in 2003, UTAUT synthesizes eight prominent technology acceptance models into a unified framework with four core determinants of usage intention and behavior. UTAUT provides a more comprehensive lens than TAM alone because it incorporates social and organizational factors that influence technology adoption. Each construct is addressed in the platform design:

a) Performance Expectancy

  • Faster Payment Processing: Multi-gateway support with instant mock payment confirmation or real Stripe/PayPal processing
  • Reliable Transactions: Database transaction safety, payment verification, stock management integrity
  • Secure Customer Experience: Visible security badges, biometric authentication, and trust indicators

b) Effort Expectancy

  • Easy Checkout Process: Gateway selection with clear descriptions, minimal clicks to complete purchase
  • Simple App Navigation: Consistent layout, clear menu structure, search functionality
  • Easy Biometric Authentication: One-click fingerprint login after initial enrollment

c) Social Influence

  • Customer Reviews: Star ratings and written reviews on all product pages
  • Testimonials: Dashboard showcases high-rated reviews from real customers
  • Social Media Integration: Share products on WhatsApp, Facebook, and Twitter
  • WhatsApp Inquiry: Direct product inquiry via WhatsApp business integration

d) Facilitating Conditions

  • Mobile Responsiveness: Full responsive design using Bootstrap 5 + Tailwind CSS
  • Cross-Platform Compatibility: Works on all modern browsers (Chrome, Firefox, Safari, Edge) and devices
  • Stable System Performance: Optimized database queries, eager loading, Vite asset bundling
  • Offline-Ready: Toast notifications and form validation work without page reloads

14. Conclusion & Recommendations

14.1 Conclusion

The SmartTrade Africa Ltd Secure E-Commerce Platform has been successfully developed as a comprehensive solution addressing the trust and security challenges faced by modern e-commerce businesses. The platform demonstrates that security and usability are not competing priorities — when properly designed, security features can enhance the user experience by providing peace of mind and reducing friction through technologies like biometric authentication.

The application successfully integrates six key technology and business domains:

  • Trusted Computing Platform (TCP) concepts including secure session management (database-backed sessions with 120-minute timeout and IP/agent tracking), device trust verification (WebAuthn credential binding to specific devices), secure token handling (CSRF tokens, signed URLs, password reset tokens, biometric setup tokens), and credential protection (bcrypt hashing, no plain-text secrets). These TCP concepts, originally developed for hardware-rooted trust in enterprise computing, have been effectively adapted to the web application context.
  • WebAuthn/FIDO2 Biometric Authentication for phishing-resistant fingerprint-based login. The implementation uses asymmetric cryptography where the private key never leaves the user's device, eliminating the risk of credential theft through server breaches or phishing attacks. The counter-based clone detection mechanism provides additional protection against credential duplication.
  • Multi-Gateway Payment Integration with Stripe, PayPal, and Mock payment processing using the Strategy Pattern. The architectural design allows new gateways (such as Flutterwave for African mobile money) to be added by implementing a single interface, ensuring the platform can adapt to evolving payment landscape requirements without code refactoring.
  • Trust Management Principles through visible trust indicators (Fingerprint Verified, Secure Payment, Trusted Session badges in the navigation bar), transparent policies (Privacy, Trust, Terms, Refund pages), customer reviews with star ratings, activity history logging, and database notifications for all critical events.
  • TAM/UTAUT Concepts ensuring the platform is perceived as useful (fast checkout, quick payment confirmation, efficient product search, helpful notifications) and easy to use (clear navigation, simple registration, one-tap fingerprint login, responsive design, consistent interface). The UTAUT constructs of performance expectancy, effort expectancy, social influence, and facilitating conditions are all explicitly addressed in the platform design.
  • Comprehensive Security following OWASP best practices including CSRF protection on every form, XSS prevention through automatic output escaping, SQL injection prevention via Eloquent ORM parameterized queries, bcrypt password hashing (12 rounds), input validation on all inputs, mass assignment protection through $fillable properties, rate limiting on authentication attempts, session regeneration after login, and path traversal prevention through the Storage facade.

The application successfully demonstrates how combining technical security measures with trust-building features creates a platform that customers can confidently use for online transactions. By making security visible and transparent, the platform addresses the fundamental trust paradox of e-commerce: users must provide sensitive information before they have established trust in the platform, so the platform must demonstrate its trustworthiness proactively.
